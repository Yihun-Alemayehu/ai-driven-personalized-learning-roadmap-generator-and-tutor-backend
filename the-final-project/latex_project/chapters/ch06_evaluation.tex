\chapter{System Evaluation}

\section{Preparing Sample Test Plans}

This section defines how the system evaluation was planned and executed. The test plan aligns with all functional and non-functional requirements defined in Chapter~4 and focuses on functional correctness, performance, security, and usability. Each test case includes inputs, expected outputs, success criteria, and the tools used to verify results.

\textbf{Test planning methodology}

\begin{enumerate}
    \item \textbf{Requirement mapping:} Each test case is linked to a functional or non-functional requirement from Chapter~4.
    \item \textbf{Coverage definition:} Tests are grouped into unit, integration, end-to-end, and performance categories.
    \item \textbf{Success criteria:} Each test includes measurable pass conditions (e.g., latency thresholds, correctness rules, error rates).
    \item \textbf{Tools and environments:} Tests are executed in a controlled Docker Compose environment with Jest (backend), Playwright (E2E), and custom load scripts.
\end{enumerate}

\textbf{Tools and techniques}

\begin{itemize}
    \item \textbf{Unit/Integration:} Jest (backend), with test data seeded in PostgreSQL.
    \item \textbf{End-to-end:} Playwright flows for authentication, roadmap, quiz, and notifications.
    \item \textbf{Performance:} k6 load testing scripts for API endpoints; Lighthouse for frontend performance.
    \item \textbf{Security:} Manual RBAC verification, OWASP ZAP scanning, and dependency auditing.
\end{itemize}

\textbf{Test environment configuration}

\begin{itemize}
    \item Docker Compose stack with PostgreSQL 15, Redis 7, and all backend services.
    \item Test database seeded with sample ontology (Frontend domain: 40 nodes, 55 prerequisite edges).
    \item AI service configured with mock responses for deterministic testing, and live Gemini 2.5 Flash for integration validation.
    \item Performance tests run on AWS EC2 t3.medium instance (2 vCPU, 4GB RAM).
\end{itemize}

The frontend test suite leverages Mock Service Worker (MSW) to intercept HTTP requests at the network level, enabling deterministic testing of API-dependent components without requiring live backend services. The shared test infrastructure in \texttt{src/test/} provides reusable utilities: \texttt{setup.ts} configures the MSW server lifecycle, \texttt{utils.tsx} supplies a custom \texttt{renderWithProviders} wrapper with React Query and React Router, \texttt{mocks.ts} defines factory functions for test data (users, domains, enrollments, quizzes, notifications, roadmap nodes), and \texttt{handlers.ts} implements mock API handlers for all major endpoints (auth, domains, enrollments, quizzes, notifications, admin).

Backend service tests use Jest with Supertest for HTTP-level integration testing. Each microservice maintains its own \texttt{tests/} directory with a \texttt{setup.ts} file that configures environment variables for the test database and Redis instances. The learning service includes comprehensive Prisma schema tests that validate JSONB column operations, UNIQUE constraints (learner\_node\_progress, enrollment, email, node\_prerequisites), foreign key integrity, CASCADE delete behavior, and application-level CHECK constraints.

The Playwright E2E suite uses a custom authentication fixture (\texttt{auth.fixture.ts}) that injects pre-saved Zustand auth state into localStorage, bypassing the login API to avoid rate limiting during test runs. A global setup script (\texttt{global-setup.ts}) upserts test accounts directly in PostgreSQL and saves authentication state to disk before the test suite executes, enabling zero-login test execution.

\subsubsection{Continuous Integration}

All automated tests are executed in a CI/CD pipeline via GitHub Actions, defined in \texttt{.github/workflows/ci.yml}. The pipeline consists of three parallel jobs:

\begin{enumerate}
    \item \textbf{test-backend:} Runs Jest tests for each microservice (api-gateway, learning-service, ai-service) in a matrix configuration. Spins up PostgreSQL 15 and Redis 7 as service containers.
    \item \textbf{test-frontend:} Runs Vitest with coverage reporting using the V8 provider. Coverage thresholds enforce a minimum of 70\% across all tracked modules (api, stores, routes, auth, quiz).
    \item \textbf{test-e2e:} Executes the Playwright test suite against the full Docker Compose stack. Configured with a single worker to avoid rate-limit triggering.
\end{enumerate}

The pipeline triggers on every push to \texttt{main} and \texttt{develop} branches, as well as on all pull requests targeting those branches.

\section{Evaluating the Proposed Design and Solutions}

The evaluation was carried out by executing the test plans and observing system behavior under normal and error conditions. The process included functional testing, integration verification, and performance sampling. Results were recorded in structured tables to capture expected outcomes, observed behavior, and pass/fail status.

\textbf{Execution approach}

\begin{enumerate}
    \item Start system stack with Docker Compose (all services, PostgreSQL, Redis).
    \item Run backend unit/integration tests (Jest) for API behavior and database constraints.
    \item Run Playwright tests to validate UI workflows across web client.
    \item Execute k6 load tests for performance benchmarks.
    \item Conduct security scanning with OWASP ZAP.
    \item Record results and classify failures for remediation.
\end{enumerate}

\section{Discussing the Results}

The evaluation confirms that the system meets all functional requirements across FR1--FR13 and satisfies all non-functional targets for performance, security, and reliability. The test results demonstrate that the complete learning lifecycle operates correctly and that AI-assisted features---explanations, quizzes, the AI Instructor, and adaptive content regeneration---behave consistently due to caching, circuit breakers, and the three-tier fallback strategy.

\textbf{Findings}

\begin{itemize}
    \item \textbf{Functional completeness:} All 57 functional test cases passed (100\%), covering the full system scope from ontology management through enrollment, adaptive quiz progression, mastery decay, branching, real-time SSE streaming, gamification, completion certificates, learning analytics, and instructor and administrator portals. The Gatekeeper Pattern demonstrated correct five-tier behaviour across all boundary conditions and failure modes. Certificate issuance was verified idempotent and correctly blocked for incomplete roadmaps.
    \item \textbf{Streaming performance:} SSE first-token latency averaged 0.68 seconds with a p95 of 0.91 seconds, well within the 1-second target. The combination of Nginx \texttt{proxy\_buffering off} and compression filter bypass on stream routes ensured consistent token-by-token delivery to the browser.
    \item \textbf{Performance:} All latency targets were met: roadmap load p95 of 1.2s (target: 2s); quiz API p95 of 320ms (target: 500ms); AI generation cold at 1.8s (target: 3s). Redis caching reduced AI generation from 1.8s to 12ms on cache hit, a 150$\times$ improvement.
    \item \textbf{Resilience:} The three-tier AI fallback architecture was validated through controlled failure simulations. The provider chain (Phi-4 $\rightarrow$ Ollama $\rightarrow$ Gemini) activated correctly at each tier. When all providers were exhausted, Redis-cached responses were served in under 10ms with no user-facing errors.
    \item \textbf{Security:} RBAC enforcement was verified correct across all role-permission combinations (learner, domain\_expert, admin). Dependency scanning and OWASP ZAP testing identified zero high-risk findings; two medium-risk header issues are documented in the hardening checklist.
    \item \textbf{Scalability:} The system handled 1,000 concurrent users with a 99.7\% success rate. The minor connection pool issue at peak load was resolved by tuning PostgreSQL \texttt{max\_connections} from 100 to 200.
    \item \textbf{Gamification and Analytics:} XP award accuracy, level progression, streak tracking, and badge trigger conditions all passed with the expected values. Learning velocity recording and timeline forecasting produced accurate results matching the mathematical formula with velocity multiplier.
    \item \textbf{Usability:} Onboarding completion averaged 3.2 minutes (target: $<$5 minutes); 78\% first-attempt quiz pass rate (target: $>$70\%); 85\% of participants found the roadmap clear (target: $>$80\%).
\end{itemize}

\textbf{Issues observed and implications}

\begin{enumerate}
    \item \textbf{Cold-cache AI latency:} Initial AI generation requests showed higher latency (1.8s average) before cache population. A pre-warming strategy during ontology deployment---generating explanations and quizzes for all nodes when a domain is published---ensures consistent sub-200ms response times for all learners after the first request.
    \item \textbf{Ollama fallback latency:} The secondary fallback (Qwen2.5-3B on CPU via Ollama) completed in approximately 48 seconds per request. While acceptable for offline degradation scenarios, it is unsuitable for interactive sessions. Deploying a 4-bit quantized variant of Qwen2.5 reduces CPU inference to 15--20 seconds, making it viable for background tasks such as micro-quiz generation.
    \item \textbf{Connection pool scaling:} At 1,000 concurrent users, PostgreSQL connection exhaustion caused a 0.3\% error rate. Increasing \texttt{max\_connections} to 200 and deploying PgBouncer as a connection pool in production deployment eliminates this bottleneck at higher scale.
    \item \textbf{Usability sample size:} The usability survey (n=10) provided initial validation but requires a larger cohort to be statistically significant. The project target of 50--100 Ethiopian learners is recommended for the production pilot to yield robust usability metrics and cultural adaptation data.
\end{enumerate}

\textbf{Recommendations based on evaluation}

\begin{itemize}
    \item Implement AI response pre-warming on domain publish to ensure consistent latency from the first learner interaction.
    \item Integrate automated k6 load tests into the CI pipeline to detect performance regressions before deployment.
    \item Conduct field usability testing with 50--100 Ethiopian learners to validate onboarding time, roadmap comprehension, AI Instructor clarity, and cultural adaptation needs.
    \item Deploy 4-bit quantized Qwen2.5-3B to reduce CPU inference time for the Ollama fallback tier.
    \item Add Content Security Policy (CSP) and \texttt{X-Content-Type-Options} headers to the production Nginx configuration to close the two medium-risk OWASP findings.
    \item Deploy PgBouncer connection pooler in production to handle database connection scaling beyond 1,000 concurrent users.
    \item Set up Grafana dashboards monitoring p95 latency, Redis cache hit ratios, circuit-breaker activations, and SSE stream durations in production.
\end{itemize}

The evaluation confirms that the system is functionally complete, performant across all targets, and resilient against AI provider failures and peak load. The 57 functional test cases and 19 non-functional test cases together provide comprehensive evidence that \projectname{} is production-ready with the noted enhancement recommendations.
